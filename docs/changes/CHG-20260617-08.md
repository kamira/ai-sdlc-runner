## Motivation
Support a multi-project flow: after launch, **register one or more project locations**; when there are
several, **designate the main project as the entry (authority)**; run a **structure analysis** across
them; and only then run the AI-SDLC four-stage loop. This maps directly onto the skill's cross-repo
model â€” an **authority source** (the main project) holds the shared contract + Guideline, each other
repo keeps a **local view + authority pointer**, coordinated changes are **XCHG** at the authority, and
`cross_repo_check.py` verifies no repo has drifted.

## Decisions & trade-offs
| Decision | Options | Why this choice |
|----------|---------|-----------------|
| Registration | interactive + persisted vs flags-only | User: interactive add + save to a reusable **workspace manifest** |
| Main project role | authority vs entry-only | User: **authority** (holds `docs/contracts/` + shared Guideline; consumers get `docs/authority.md` pointer) â€” matches the skill |
| Structure analysis depth | full four structures + cross-repo vs light scan | User: **full** â€” per-repo four structures + cross-repo contract surface; runner does the deterministic scan/scaffold, the A1 stage fills the analytical content |
| Consistency | none vs cross_repo_check gate | User: **persist + `cross_repo_check` as a gate** in the loop; one **integration acceptance** at close-out |
| Where the manifest lives | runner vs authority | At the **authority** (`<authority>/.sdlc-workspace.json`) â€” travels with the source of truth |
| Executor working dir | unset vs per-project cwd | Thread the target repo as the agent's **cwd** (needed so a CLI agent works on the right repo) â€” closes the earlier gap |

## Modification Guide
### Goal
Add a workspace concept (authority + repos, persisted), an interactive registration step, a structure-
analysis pass (scan + scaffold four structures + set up authority contract & pointers), and a workspace-
aware run that gates on `cross_repo_check` and closes with an integration acceptance. Thread the target
repo as the executor's working directory.
