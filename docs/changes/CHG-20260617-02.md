# CHG-20260617-02 — Interactive arrow-key menu for the runner CLI

- Project: ai-sdlc-runner
- Date: 2026-06-17
- Type: new feature
- Proposed by: Harutsuki (wants a selectable list UI, like the Claude CMD TUI screenshot)
- Implemented by: I1 (lead implementer, this session)
- Risk: low  (additive CLI convenience layer; no governance logic, no contract surface touched; dispatches to existing commands)
- Autonomy: (none — default halt policy applies)
- Related: FR-12 (CLI subcommands); builds on CHG-20260617-01

## Motivation
Offer an interactive, arrow-key selectable menu (like the referenced "Claude CMD" TUI) so users can
pick operations without memorizing subcommands. This lives entirely in the **runner's own CLI
surface** — which is *not* part of the locked contract (the contract is the skill's stable output) —
so adding it is permitted (build-guide §1.7: runner-side interface may evolve; don't touch the
contract).

## Decisions & trade-offs
| Decision | Options | Why this choice |
|----------|---------|-----------------|
| TUI implementation | stdlib `curses` vs third-party (questionary/prompt_toolkit) | User chose stdlib curses — keeps the runner zero-dependency (build-guide stdlib-only stance); third-party would add a runtime dep |
| Non-TTY behavior | fail vs numbered fallback | Fallback to a numbered prompt when stdin/stdout isn't a TTY or curses is unavailable → works in pipes/CI/Windows |
| Invocation | subcommand only vs no-arg vs both | User chose **both**: bare `runner` opens the menu, and an explicit `runner menu` also exists; existing `run`/`migrate`/`status` keep their non-interactive behavior |
| Menu actions | new capabilities vs map to existing | Map only to existing run/migrate/status (+ Help/Exit) — acceptance aligns to source; no new governance-bearing behavior smuggled in |

## Modification Guide
### Goal
Add an interactive menu to the CLI that dispatches to the existing commands, with zero new
dependencies and a non-TTY fallback. Map to FR-12 (CLI surface).

### Impact scope
| Affected item | Type | Change | Knock-on effect |
|---------------|------|--------|-----------------|
| `src/ai_sdlc_runner/tui.py` | directory/logical/design | NEW module: `select()` (curses arrow menu + numbered fallback), `_parse_choice()` | Pure UI helper; no governance logic |
| `src/ai_sdlc_runner/cli.py` | logical/design | `run`/`migrate`/`status` unchanged; subparsers no longer `required`; bare `runner` → menu; add `menu` subparser; `cmd_menu()` loop dispatches to existing `cmd_*` | Backward compatible (existing invocations behave identically) |
| `docs/structure/{directory,logical,design}.md` | structure | Add the `tui` module/component | Keep structure docs current |
| `README.md` | docs | Document the menu (EN + 繁中) | — |
| `tests/test_tui.py` | logical | NEW: cover `_parse_choice` + fallback selection | Acceptance evidence |

### Modification steps
1. Add `tui.py` with `select(title, options, ...)` and pure `_parse_choice(raw, n)`.
2. In `cli.py`: make subparsers optional; add `menu` subparser; implement `cmd_menu()` that selects an action, prompts for project/version/risk, and calls the existing `cmd_run`/`cmd_migrate`/`cmd_status`.
3. Sync structure docs; document in README (EN + 繁中); add `tests/test_tui.py`.

### Risk & rollback
- Risk: low — additive convenience; the menu only collects inputs and calls existing commands, so all

- Rollback: delete `tui.py`, revert the `cli.py` diff; `run`/`migrate`/`status` are unchanged.
