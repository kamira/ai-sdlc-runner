# CHG-20260617-04 — Detect skill updates at the local skill location

- Project: ai-sdlc-runner
- Date: 2026-06-18
- Type: new feature
- Proposed by: Harutsuki (ai-skills now provided locally with tags ai-sdlc-v1.0.0 / ai-sdlc-v1.1.0; runner should detect whether the local skill has an update)
- Implemented by: I1 (lead implementer, this session)
- Risk: low  (read-only detection; reuses existing version-read + lock; no governance logic, no auto-migrate)
- Autonomy: (none — default halt policy applies)
- Related: FR-1/FR-3/FR-4/FR-5 (version read, lock, migrate semantics); CHG-20260617-01

## Motivation
The `ai-skills` repo is now provided locally and carries two contract versions (`ai-sdlc-v1.0.0`,
`ai-sdlc-v1.1.0`; v1.1.0 adds a role-responsibility catalog). The runner should **detect** when the
local skill location is newer than what a project is locked/expected at, and classify the gap
(patch = auto, minor/major = migrate required) — surfacing it instead of silently drifting. Detection
is read-only; it never auto-migrates (the validating `migrate` stays explicit, per §5).

## Decisions & trade-offs
| Decision | Options | Why this choice |
|----------|---------|-----------------|
| Compare against what | lock-only vs expected vs both | Compare the local skill version to the **project lock** when given, else the config-expected version; covers both "project behind" and "no project" cases |
| Newer-version signal | SKILL.md only vs also git tags | Also scan the skill repo's version tags (`ai-sdlc-vX.Y.Z` / `vX.Y.Z`) so a pinned-but-outdated submodule is flagged even before checkout changes |
| Classification source | re-derive vs reuse contract_key | Reuse `contract_key` + the §5 patch/minor/major rules — no new policy duplicated |
| Action on minor/major | auto-migrate vs report only | Report + tell the user to run `migrate` (validating upgrade stays explicit; never auto-cross a boundary) |
| Tag name | `v1.0.0` vs `ai-sdlc-v1.0.0` | The skill repo renamed tags to skill scope; update `.gitmodules` guidance accordingly |

## Modification Guide
### Goal
Add read-only detection of skill updates at the local skill location and surface it in the CLI,
status, dashboard, and menu. Reuse existing version/lock logic.

### Impact scope
| Affected item | Type | Change | Knock-on effect |
|---------------|------|--------|-----------------|
| `src/ai_sdlc_runner/contract.py` | logical/design | Add `UpdateInfo`, `detect_update(skill_path, expected=None, project_dir=None)`, `available_version_tags(skill_path)` | Read-only; reuses `read_skill_version`/`contract_key`/`read_lock` |

### Modification steps

2. `cli.py`: `check [project]` subcommand; status line; menu action.
3. `dashboard.py`: optional skill line in the Status panel.
4. Update `.gitmodules` tag name; sync structure docs; README (EN + 繁中); tests.
