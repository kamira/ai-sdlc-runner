# CHG-20260617-06 — Pluggable, platform-agnostic executor backends (API or subscription)

- Project: ai-sdlc-runner
- Date: 2026-06-18
- Type: new feature
- Proposed by: Harutsuki (do not lock to any AI platform; support running agents via API or via a subscription/CLI)
- Implemented by: I1 (lead implementer, this session)
- Risk: medium  (introduces outbound API calls + external command execution as runtime options; gates unchanged; secrets via env only)
- Autonomy: (none — default halt policy applies)
- Related: CHG-20260617-01 (orchestrator `agent_executor` injection); build-guide §1.7 (runtime isolation)

## Motivation
The agent-execution backend is a **runtime concern**, not part of the contract (build-guide §1.7:
"the contract targets the skill's stable output, not Claude Code's current behavior; runtime changes →
change the runner, not the contract"). Make it pluggable so the runner can drive agents through **any**
AI platform — an HTTP **API** (with a key) or a **subscription / local CLI agent** — without being tied
to one vendor.

## Decisions & trade-offs
| Decision | Options | Why this choice |
|----------|---------|-----------------|
| Where the backend lives | contract vs runtime/config | Runtime, config-driven (§1.7); the skill/contract is untouched |
| Backends | one vendor vs stub + command + api | `stub` (offline/dry-run), `command` (any subscription/CLI agent), `api` (HTTP) — platform-agnostic |
| API shape | one provider vs adapters | Generic config (`base_url`, `model`, `api_key_env`) + `anthropic`/`openai`/`generic` request+parse adapters |
| HTTP client | requests/httpx vs stdlib | `urllib` (stdlib) — keeps the zero-dependency stance |
| Secrets | config vs env | API keys are read from an **environment variable** named in config, never stored in config/repo |
| Default | api vs stub | `stub` by default (safe, offline); the real backend is opt-in via config or `--backend` |
| Gate coupling | none | The executor only does agent work; halt gates and red-line stops are unchanged and independent |

## Modification Guide
### Goal
Add an executor abstraction with stub/command/api backends, config-driven and stdlib-only, injected
into the existing `orchestrator.run(agent_executor=...)`. No change to gating.

### Impact scope
| Affected item | Type | Change | Knock-on effect |
|---------------|------|--------|-----------------|
| `src/ai_sdlc_runner/executors.py` | NEW logical/design | `StubExecutor`, `CommandExecutor`, `ApiExecutor`, `build_request`/`parse_response`, `from_config` | Runtime backend; no governance logic |
| `src/ai_sdlc_runner/cli.py` | logical | `run --backend {stub,command,api}`; build executor from config; menu choice | Default stub → behavior unchanged |
| `config/runner.yaml` | data | Add `executor:` block (backend + command/api settings) | Opt-in; secrets via env |
| `docs/structure/*` , `README.md` | docs | Document backends | — |
| `tests/test_executors.py` | logical | NEW: stub/command(local echo)/api request+parse/factory | Acceptance evidence |

### Modification steps
1. `executors.py`: the three backends + `build_request`/`parse_response` (pure, testable) + `from_config`.
2. `cli.py`: build the executor from config (or `--backend`); pass its `run` to the orchestrator (stub → keep current path).
3. `config/runner.yaml`: add the `executor` block (documented; default `stub`).
4. Sync structure docs + README (EN + 繁中); add tests.

### Structure document updates
- [ ] docs/structure/directory.md: add `executors.py`
- [ ] docs/structure/logical.md: add `executors` module + backend selection
- [ ] docs/structure/design.md: executor contract + `build_request`/`parse_response`

### Risk & rollback
- Risk: medium — outbound HTTP and local command execution become possible, but only when the user
  opts in via config/`--backend`; keys come from env; **gates and red-line halts are unchanged** (the
  executor runs agent work, not stage gating). Default stays `stub`.
- Rollback: remove `executors.py`, the `--backend` flag, and the `executor` config; orchestrator falls
  back to the internal stub.

## Structure change summary
Adds a runtime-isolated, config-driven executor layer (stub/command/api) injected into the existing
orchestrator hook. Platform-agnostic; stdlib-only; secrets via env; gates untouched. Structure docs
updated in the same change.
